# 2026 Τελική Εργασία: Ενσωματωμένα Συστήματα Πραγματικού Χρόνου

## Στόχος: Υλοποίηση πολυνηματικού ενσωματωμένου συστήματος πραγματικού χρόνου σε περιβάλλον Linux (Raspberry Pi Zero W ή όποιο άλλο έχετε στη διάθεσή σας) για την ασύγχρονη λήψη, επεξεργασία και σύγχρονη καταγραφή δεδομένων τηλεμετρίας από ένα παγκόσμιο κανάλι ροής δεδομένων (Firehose).

## 1. Περιγραφή του Data Stream (Target)

Για τις ανάγκες της εργασίας, θα συνδεθείτε στο WebSocket API του Bluesky Jetstream Firehose. Πρόκειται για μια ζωντανή ροή δεδομένων (δημόσιες δημοσιεύσεις, likes, follows) από το αποκεντρωμένο δίκτυο AT Protocol, μεταφρασμένη σε πραγματικό χρόνο σε μορφή κειμένου JSON.

**Endpoint:** wss://jetstream1.us-east.bsky.network/subscribe?wantedCollections=app.bsky.feed.post

**Μορφή Δεδομένων:** Κάθε μήνυμα είναι ένα JSON object. Το σύστημά σας πρέπει να αναγνωρίζει και να κατηγοριοποιεί το πεδίο "kind" του μηνύματος.

## 2. Αρχιτεκτονική Συστήματος & Απαιτήσεις Νημάτων

Η εφαρμογή θα υλοποιηθεί αποκλειστικά σε γλώσσα C (User Space) στο Raspberry Pi (Zero W ή όποιο άλλο έχετε στη διάθεσή σας), χρησιμοποιώντας τη μεθοδολογία Producer-Consumer (Παραγωγός-Καταναλωτής) με POSIX Threads (pthreads), mutexes (pthread_mutex_t) και condition variables (pthread_cond_t).

Η αρχιτεκτονική αποτελείται από τρία (3) βασικά νήματα:

### Α. Νήμα 1: Ο Παραγωγός (Producer - Ασύγχρονο / Δικτυακό)

**Λειτουργία:** Συνδέεται στο WebSocket endpoint χρησιμοποιώντας μια ελαφριά βιβλιοθήκη C για WebSockets (προτείνεται η libwebsockets https://libwebsockets.org).

**Συμπεριφορά:** Λειτουργεί ασύγχρονα (event-driven). Κάθε φορά που φτάνει ένα νέο raw text frame (JSON) από το δίκτυο, το ανακτά και το τοποθετεί αμέσως σε έναν κυκλικό buffer σταθερού μεγέθους (Bounded Circular Queue), ξυπνάει τον καταναλωτή μέσω condition variable και επιστρέφει αμέσως στο δίκτυο για να μην χαθούν πακέτα.

### Β. Νήμα 2: Ο Καταναλωτής (Consumer - Event-Driven)

**Λειτουργία:** Ξυπνάει μόλις ο Παραγωγός τοποθετήσει δεδομένα στον κυκλικό buffer.

**Συμπεριφορά:** Αφαιρεί το JSON string από τον buffer, το κάνει parse χρησιμοποιώντας έναν ελαφρύ JSON parser (προτείνεται η cJSON ή η jsmn) και αναγνωρίζει το πεδίο "kind".

**Επεξεργασία:** Αυξάνει τις αντίστοιχες καθολικές (global) μεταβλητές-μετρητές, οι οποίες προστατεύονται από mutex, για τους 4 τύπους μηνυμάτων:

1. commit
2. identity
3. account
4. info (μηνύματα συστήματος/σφαλμάτων)

**Προσοχή:** Για λόγους απόδοσης, απαγορεύεται η εκτύπωση στην οθόνη (printf) ή η εγγραφή σε αρχείο μέσα σε αυτό το νήμα.

### Γ. Νήμα 3: Ο Καταγραφέας / Monitor (Synchronous / Periodic)

**Λειτουργία:** Εκτελείται αυστηρά περιοδικά, κάθε ένα (1) δευτερόλεπτο, στο δευτερόλεπτο. Πρέπει να υλοποιηθεί με POSIX Timers (timer_create) ή ακριβές clock_nanosleep() (όχι απλό sleep()), ώστε να αποφευχθεί η ολίσθηση του χρόνου (clock drift).

**Συμπεριφορά:** Μόλις ξυπνήσει, "κλειδώνει" προσωρινά το mutex, λαμβάνει τους μετρητές των μηνυμάτων του τελευταίου δευτερολέπτου, μηδενίζει τους μετρητές για το επόμενο δευτερόλεπτο, υπολογίζει την τρέχουσα πληρότητα του κυκλικού buffer (%) και την τρέχουσα χρήση της CPU (%).

**Εγγραφή:** Αποθηκεύει τα δεδομένα αμέσως (append) σε ένα ενιαίο αρχείο κειμένου με όνομα metrics_log.txt.

## 3. Προδιαγραφές Αρχείου Καταγραφής (Log File)

Κάθε γραμμή στο αρχείο metrics_log.txt πρέπει να είναι σε μορφή CSV (Comma-Separated Values) και να περιέχει ακριβώς τα εξής πεδία:

```text
Seconds,Nanoseconds,Commit_Count,Identity_Count,Account_Count,Info_Count,Buffer_Occupancy_Pct,CPU_Pct
```

### Κανόνες Πεδίων:

**Χρονική Σήμανση (Timestamp ):** Πρέπει να καταγράφεται η ακριβής χρονική στιγμή εκτέλεσης της καταγραφής με χρήση της clock_gettime(CLOCK_REALTIME, &ts). Θα αποθηκεύεται σε δύο ξεχωριστά πεδία: ο ακέραιος αριθμός δευτερολέπτων Unix (ts.tv_sec) και το κλάσμα νανοδευτερολέπτων (ts.tv_nsec).

**Τετράδα Μηνυμάτων (Quad Tuple):** Ο αριθμός των μηνυμάτων ανά τύπο που λήφθηκαν στο συγκεκριμένο παράθυρο του 1 δευτερολέπτου.

**Πληρότητα Buffer (%):** Το ποσοστό των θέσεων του κυκλικού buffer που ήταν κατειλημμένες τη στιγμή της καταγραφής.

**Χρήση CPU (%):** Το ποσοστό χρήσης της CPU του συστήματος κατά το τελευταίο δευτερόλεπτο. Για να γίνει αυτό με το ελάχιστο δυνατό overhead, το νήμα θα διαβάζει το εικονικό αρχείο /proc/stat (την πρώτη γραμμή cpu ), υπολογίζοντας τη διαφορά (delta) των jiffies (συνολικών και idle) μεταξύ της τρέχουσας και της προηγούμενης ανάγνωσης.

## 4. Κριτήρια Απόδοσης Πραγματικού Χρόνου (Real-Time Metrics)

Στόχος είναι η ελαχιστοποίηση της καθυστέρησης (jitter) και της ολίσθησης (drift) του περιοδικού νήματος.

**Καθυστέρηση (Jitter):** Η απόλυτη διαφορά του πραγματικού χρόνου αφύπνισης του Καταγραφέα από την ιδανική προθεσμία (ακριβώς 1.000000000 δευτερόλεπτο από την προηγούμενη).

**Ολίσθηση (Drift):** Η συσσώρευση σφαλμάτων χρονισμού που θα είχε ως αποτέλεσμα το πρόγραμμα να "χάσει" δευτερόλεπτα κατά τη διάρκεια των 24 ωρών.

## 5. Παραδοτέα Εργασίας

### Α. Αναφορά (Έως 4 Σελίδες)

Η αναφορά πρέπει να περιέχει:

Αναλυτική περιγραφή της αρχιτεκτονικής του κώδικα (διαχείριση νημάτων, συγχρονισμός, μέγεθος κυκλικού buffer, αντιμετώπιση race conditions).

Μετα-επεξεργασία (Post-processing) σε MATLAB/Python/etc: Φορτώστε το αρχείο metrics_log.txt στο MATLAB και δημιουργήστε τα εξής διαγράμματα:

* Διάγραμμα Jitter: Τη χρονική διαφορά (θετική ή αρνητική σε milliseconds) της εκτέλεσης του περιοδικού νήματος από το ιδανικό δευτερόλεπτο σε συνάρτηση με τον χρόνο.
* Διάγραμμα Φόρτου & Buffer: Διπλός άξονας με το πλήθος των μηνυμάτων (Hz) και το ποσοστό πληρότητας του κυκλικού buffer, για την ανάδειξη της "εκρηκτικότητας" (burstiness) του δικτύου.
* Διάγραμμα CPU: Τη συσχέτιση του ρυθμού εισερχόμενων μηνυμάτων (Hz) με το ποσοστό (%) που η CPU μένει αδρανής (Idle) ή απασχολημένη.

Σχόλια και παρατηρήσεις που αποδεικνύουν ότι το σύστημά σας μπορεί να λειτουργήσει για μεγάλα χρονικά διαστήματα (ημέρες) χωρίς απώλειες δεδομένων (memory leaks), σφάλματα segmentation faults, και πώς συμπεριφέρεται αν υπάρξει στιγμιαία αποσύνδεση στο δίκτυο (network resilience).

### Β. Κώδικας & Δεδομένα (Σύνδεσμος στο GitHub / Cloud)

Το αρχείο δεδομένων (metrics_log.txt) που συλλέχθηκε συνεχόμενα για ακριβώς 24 ώρες τη Τρίτη 30/06/2026 (00:00:00 - 23:59:59).

**Σημείωση:** Δε θα γίνουν δεκτά αποτελέσματα από προσομοίωση ή εκτέλεση σε άλλο υπολογιστή. Το πρόγραμμα πρέπει να τρέξει στο physical hardware του Pi.

## 6. Οδηγίες & Προετοιμασία

Πριν ξεκινήσετε τη συγγραφή του κώδικα σε C, προτείνεται να "ανιχνεύσετε" τη συμπεριφορά, τη δομή JSON και τον ρυθμό (Hz) του Jetstream Firehose χρησιμοποιώντας το MATLAB.

Μπορείτε να χρησιμοποιήσετε τη γέφυρα Python του MATLAB (pyenv) για να ανοίξετε ένα δοκιμαστικό socket, να πειραματιστείτε με τη συνάρτηση jsondecode και να δείτε πώς μοιάζουν τα πεδία commit, identity και account, επιβεβαιώνοντας έτσι τις προδιαγραφές των δεδομένων πριν τις αποτυπώσετε στη C.
